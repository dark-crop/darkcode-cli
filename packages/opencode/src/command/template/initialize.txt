Analyze this repository and create (or improve) an `AGENTS.md` at `${path}` that a future darkcode
session reads first, the same way it reads `CLAUDE.md`. Investigate with your tools before writing -
do not guess. When done, briefly tell the user what you wrote and why.

Extra focus from the user (honor if present): $ARGUMENTS

Investigate, executable sources first (they win over prose when they disagree):
- README, root manifests + lockfiles, workspace/monorepo config
- build / test / lint / format / typecheck / codegen scripts, CI workflows, pre-commit/task-runner config
- existing agent rules to fold in: `AGENTS.md`, `CLAUDE.md`, `.cursor/rules/`, `.cursorrules`,
  `.github/copilot-instructions.md`, repo-local `opencode.json`
- then, only if the architecture is still unclear, read the few files that show how the system is
  wired (real entrypoints, package boundaries, execution flow) - not random leaf files

Write AGENTS.md to be SHORT and high-signal. The test for every line: "would a competent agent get
this wrong or waste time without it?" If not, cut it. Capture the hard-won facts:
- exact commands the agent would otherwise guess wrong - build, run, lint, typecheck, and especially
  how to run ONE test / one package / a focused check; note required order (e.g. codegen -> build)
- monorepo boundaries and what actually owns each major directory + the real entrypoints
- toolchain quirks: generated code, migrations, env loading, dev servers, deploy flow, build artifacts
- conventions that DIFFER from language/framework defaults, and constraints worth preserving from the
  existing rules files above

Do NOT include: generic engineering advice, obvious language conventions, exhaustive file trees,
tutorials, or anything you could not verify from the repo.

If `AGENTS.md` already exists, edit it in place: keep what's still true, delete stale or fluffy lines,
reconcile it with the current code. Keep it compact - a simple repo gets a simple file. Ask the user a
question ONLY for something important the repo genuinely cannot answer (undocumented conventions,
release/branch expectations, hidden setup prerequisites); otherwise just write the file.
